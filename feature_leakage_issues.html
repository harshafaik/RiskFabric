RiskFabric Documentation

While designing a fraud detection model for flagging potentially fraudulent transactions using XGBoost, some problems were discovered that made it unsuitable for being used in real-time scoring. The issue revolves around an individual feature primarily deciding the model's decision making.

Transactions were synthetically generated for 10,000 customers with a total figure being around 6M transactions. After feature engineering and interpolating them into a gold_master_table which is used by XGBoost for training. An AUC score of 0.9079 was achieved which felt more realistic than the previous test which ran on a smaller dataset (4.3M transactions and achieved an AUC score of 0.97). However, the crux of the issue became apparent when the top features by importance were checked:

What this means here is that if this model looks at a transaction and sees that the amount seems fishy, there's a very high chance that it'll flag this transaction as fraud, without even considering the other characteristics of it. This isn't optimal since the model is supposed to take all the constraints into consideration such as temporal factors, geographic behavior and many more. As of now, the model is solely relying on amount as the primary classifier for deciding whether to flag a transaction or not. To combat this, we can use two strategies, one we could remove amount as a training flag to see how the model performs when forced to rely solely on behavioral flags. The other option is that we use feature binning, which lets us reduce the model's reliance on exact values. After testing option 2 and seeing negligible difference in terms of feature importance, option 1 was evaluated, which revealed the underlying issue within the system. After removing amount as a feature and executing the training script again, a ROC score of 0.5868 was achieved—a significant decrease from the previous result. However, the resulting feature importance distribution was more revealing:

This indicates that the behavioral features designed have very little predictive power to be captured by the model. Essentially, the model can't distinguish them from normal variation. Re-tuning the fraud generator is required to create distinctive behavioral signals and ensure those features are engineered effectively and processed through to the gold table. This ensures each fraud signature has its distinctive characteristics so that it can be captured by the model.

Analysis of the pipeline identified a significant gap: The behavioral features engineered in Rust were being dropped before reaching the XGBoost model:

  1. The "Silently Dropped" Features In src/etl/features/sequence.rs, high-value signals are calculated that would likely address the 0.58 AUC problem, but they are missing from the ClickHouse tables:

    • amount_deviation_z_score: This measures if ₹5,000 is "normal" for that specific customer. Without this, the model only sees the absolute ₹5,000 and assumes it's fraud because the average transaction is ₹500.
    • fraud_type & campaign_id: These are currently calculated but not stored in the Silver layer.
    • Granular Anomalies: geo_anomaly, device_anomaly, and ip_anomaly are being calculated in the transformation but are not being selected in the final Gold table join.

  2. The Gold Table Join is too narrow The run_gold_master function in src/bin/etl.rs only pulls a small subset of columns from the Silver tables. It's ignoring the very features needed to replace the "Amount Shortcut."

The Plan to Fix It:

Step 1: Repair the ETL Pipeline (The "Plumbing" fix) * Update the CREATE TABLE statement for fact_transactions_silver to include the missing behavioral columns.

  • Update the run_gold_master query to pull these features into the final training set.
  • Outcome: The model will finally "see" the Z-Score and the behavioral context.

Step 2: Re-tune the Generator (The "Signal" fix) * The fraud_rules.yaml configuration is modified to ensure fraudulent transaction amounts overlap with legitimate amounts. * Outcome: The model is forced to stop using "High Amount" as a shortcut and instead utilize the "Z-Score" and "Velocity" features fixed in Step 1.

After retuning the generator to create more overlap and amplify behavioral signals, the score dropped slightly to 0.7960, fraud amounts now blend into legitimate ranges, so amount_deviation_z_score has less to work with. The model is being forced away from the amount shortcut. AUC dropped because the problem genuinely got harder.

However, amount_deviation_z_score is still at 93% despite the overlap. This means the Z-score is still capturing enough separation between fraud and legitimate amounts to dominate. The overlap wasn't aggressive enough. The problem appears to be fraud amounts as they are mostly specific values — ₹5000, ₹8500, ₹12000. Legitimate transactions cluster around ₹660. Due to this, the Z-score still sees as "this customer normally spends ₹660, this transaction is ₹8500 — suspicious." The relative deviation is still huge.

The Z-score will only become less dominant when fraudsters transact at amounts that are normal for that specific customer. That requires the generator to look up the customer's monthly_spend and generate fraud amounts within their normal range:

account_takeover:
amount_strategy: "customer_normal_range"  # instead of fixed high_value_amounts
amount_multiplier: 0.8_to_1.2  # within customer's normal band

This makes fraud amounts become customer-relative rather than absolute. In this iteration, the Z-score was removed and the model was trained without amount_deviation_z_score and escalating_amounts_flag to evaluate the strength of the pure behavioral signals.

The score dropped to 0.5572, and for the first time, feature importance is evenly distributed. No single feature above 17%. Every behavioral signal contributing meaningfully. This is what a healthy feature set looks like structurally.

So the problem isn't with the features, it's that none of them have enough signal strength to detect fraud reliably. The model is genuinely confused because fraudsters in the simulation behave too similarly to legitimate customers. For instance:

  • transaction_channel at 17% — channel bias exists but weak. Fraudsters use online/UPI slightly more but not enough to be distinctive.
  • cf_night_tx_ratio at 15% — night patterns exist but fraud isn't concentrated enough at night to stand out.
  • rapid_fire_transaction_flag at 8.5% — velocity fraud is happening but not firing consistently enough.
  • merchant_category_switch_flag at 3.25% — almost no signal. Fraudsters are shopping at similar merchants as legitimate customers.

To fix this at the root level, the code responsible for injecting the fraudulent signatures and behaviors needs to be looked into, so that their signal strength could be increased for training genuinely honest, behaviorally-driven fraud model.

In this project, fraud.rs is primarily responsible for injecting fraud labels as well as altering the fraudulent transactions' behavior according to the fraud signature. There are two configs present for driving different behavior for the transactions, geo_anomaly_prob and device_anomaly_prob. Inspecting geo_anomaly_prob showed some significant limitations:

As of now, if a transaction had the bool flag geo_anomaly as true, the transaction's latitude and longitude with completely random coordinates from the global range. While this creates an anomaly, it doesn't provide a behavioral signal that the model can learn unless the model has access to the user's "Home" coordinates or a feature like "Distance from Home." Due to this, the model only sees final_lat and final_lon. Since legitimate transactions are also spread across India (just clustered around their specific home), a random coordinate in India looks "normal" to a model that doesn't know where the customer lives.

To fix this, a new feature, Spatial Velocity is introduced in the ETL layer. This would measure: distance(txn_N, txn_N-1) / time(txn_N, txn_N-1). This will help the model catch instances of high-velocity spatial anomalies, such as someone transacting in Delhi and then 10 minutes later in Bangalore for example.

The AOC jumps to 0.6868 from a single feature addition. spatial_velocity at 61% is now the dominant behavioral feature — and unlike amount, this is a genuinely behavioral signal. A transaction physically impossible to make given travel time is real fraud evidence, not a shortcut. This also had a cascading effect on other features:

  • time_since_last_transaction from 16% → 6.52%

There are problems that still needed to be addressed:

  • spatial_velocity at 61% is still too dominant. It's capturing almost all the geo_anomaly fraud signal in one feature. The current geo_anomaly implementation currently teleports fraudsters to random India coordinates — every ATO transaction fires the impossible travel flag almost guaranteed.
  • cf_night_tx_ratio dropped to 1.28% — nearly invisible. Night behavior is not distinctive enough in your generator.
  • card_present still at 0% — CNP fraud isn't being captured at all.

Seeing cf_night_tx_ratio at 1.28% led to checking the hourly distribution of transactions and more specifically under Account Takeover (ATO) fraud, which showed that the hourly_weights currently peak in the very early morning (00:00 - 04:00) and again late in the evening (18:00 - 23:00). This realistically simulates when attackers are most active or when victims are less likely to notice alerts.

However, seeing the low percentage suggests that the "Night Ratio" isn't a strong enough signal, likely because either:

  1. Legitimate users also spend at night (e.g., late-night food or online orders).
  2. The ATO peak isn't "sharp" enough compared to the legitimate baseline.

This was addressed by updating the account_takeover hourly weights to concentrate 70%+ of transactions between 00:00 and 04:00, creating a sharp contrast with the peak legitimate spending period (10:00 - 22:00). A new feature named hour_deviation_from_norm was added in the ETL layer to capture this anomaly at the transaction level.

One more feature was implemented named hour_deviation_from_norm. This calculated each customer's average transaction hour and then determining the absolute deviation for each specific transaction, providing a strong per-transaction signal for temporal anomalies.

After wiring 'CNP' back, it jumped to 5%. However, rapid_fire_transaction_flag disappeared from the top 10. Velocity abuse fraud should be firing this heavily. If it's gone it means either velocity fraud isn't clustered tightly enough or the flag threshold is too high.

Looking at the code, this particular flag currently uses a threshold of 300 seconds (5 minutes). Also, the problem extended with max_interval_seconds for velocity_abuse set to a random minute (0-59) within a selected hour. For fraud signatures that depend on seconds-level timing — velocity abuse especially — this is too coarse.

A more realistic temporal pattern, especially for fraud bursts was implemented ensuring that transactions occur in much tighter sequences (e.g., seconds apart) when a burst or velocity abuse is triggered. max_burst_interval_seconds is used control this behavior. Moreover, if a transaction is part of a "burst" (like sequential_takeover or coordinated_attack), its timestamp will be just seconds away from the previous one, rather than a random minute within the hour. This will create a much sharper behavioral signal for the rapid_fire_transaction_flag to capture. A new feature named hour_deviation_from_norm was added as well.
